# Writeup-Home Robot LLM

## Approach

The original `agent.py` sends the entire command to the model at once. It gets back a complete plan and executes it without checking first. This doesn’t meet the requirements in the README. A model can imagine an object or location that doesn't exist, can't know what’s actually in the apartment before checking, and can’t respond if a `pick()` fails while following a multi-step plan.

I replaced the “plan once, execute all steps” method with a closed-loop **ReAct-style control loop**. In each turn, the model or a backup reasoner suggests **one** next action based only on what the robot has sensed. The agent checks that action, carries it out, and then feeds the real outcome back before asking for the next step. This approach directly addresses the four points requested in the README:

- **Grounding** - the model only sees locations from `robot.known_locations` (always valid) and objects from `robot.known_objects` (only listed once the robot has been there and sensed them). It can’t reference an object it hasn’t seen because it isn't part of its context yet. The simulator also rejects unknown locations or objects with a clear `Result`, and that result is included in the next prompt.
- **Safety** - objects have a `category` field from sensing. I treat `sharp_utensil` and `medication` as safety-sensitive and block them in two ways: in the system prompt (rule 3), and - because I don’t want to rely on the model’s compliance for this - as a **hard stop in code** that intercepts any `pick()` on such objects and requires a `speak()` instead before execution can happen. Medication is outright declined (dosage and identity cannot be confirmed with one question); for a sharp object, I ask a clarifying question about intent.
- **Ambiguity** - if a vague request ("something to drink") applies to more than one sensed object, the loop asks which one instead of making a guess. This determination only becomes possible after sensing, which the loop naturally handles: navigate to a likely spot, sense, then check for ambiguity.
- **Recovery** - `pick()` failures appear in the feedback history, so the agent can retry (limited to 3 attempts) and, if the gripper continues slipping, tell the person honestly rather than pretending success or getting stuck in a loop. A strict `MAX_STEPS` cap (10) serves as the final safeguard against any runaway loop.

## Two "brains" behind the same loop

`call_llm()` first attempts to use a real LLM - Anthropic, Groq, Gemini, then a local Ollama server, whichever has a key or endpoint available from environment variables (`ANTHROPIC_API_KEY`, `GROQ_API_KEY`, `GEMINI_API_KEY`, or a local Ollama instance). If none are set up or a call fails, the loop switches to `heuristic_action()`, a small deterministic reasoner that follows the same rules using keyword and category matching instead of a model call.

This was a purposeful choice, not a shortcut: the *control loop* - grounding checks, safety stops, retry limits, and step caps - is responsible for the safety-related work and encompasses whichever brain proposes actions. Having a working fallback allowed me to develop and test the entire decision process, including more difficult edge cases (repeated grasp failures, an object that likely doesn’t exist, an out-of-scope request) entirely offline and observe precisely how it performs, instead of troubleshooting against a live API with rate limits. I ran `python run.py --test` on all scenarios in `tests.py` this way (`--fail 0.3` and to test recovery more rigorously, `--fail 1.0` to force every grasp to fail) - with no API key set, so this is entirely the fallback path - and confirmed the behavior in the "Tradeoffs" section below. When a real key is set, the same loop, safety stops, and grounding rules apply; only the source of the suggested action changes.

## Tradeoffs

- **Destination unknown.** The simulator cannot determine where the person is standing; there’s no sensor for it and no skill to query it. "Bring me X" is therefore unclear about *where* "me" is located. I set the default to `living_room` as a common gathering spot, which is what the original mock implementation also suggested. A more accurate version might have the robot ask "where should I bring it?" the first time, but that could result in unnecessary chatter for the common situation. I noted this as a known limitation instead of ignoring it.
- **Asking vs. just delivering with a caution for sharp objects.** For the knife case, I chose to ask before fetching it rather than delivering it with a spoken warning. Asking is more conservative and seems more reasonable with no context about the request, but a "here it is, be careful" response could also work and may create less friction for a legitimate cooking request. I would want real usage data to find out which option people prefer less.
- **Heuristic fallback is keyword-based,** so it’s noticeably less flexible than a real model at understanding phrasing it wasn't designed for (for example, it won't grasp "I could use some hydration" like an LLM would). It’s meant as a backup for offline development and testing without a key, not as the main path - I expect the real LLM path to handle novel phrasing more effectively while still following the same code-level safeguards.
- **Safety category list is short** (`sharp_utensil`, `medication`). A real deployment would need a broader list, probably maintained externally (cleaning products, other harmful objects) instead of just two hardcoded categories.

## what I’d do differently with more time

- The simulator does not update an object’s recorded location after `place()`, so if a task ever needs to re-locate something *after* delivering it in the same run, the agent would have outdated information. I managed this by detecting "we already placed something and we're not holding anything" as a completion signal rather than trying to re-detect location. A cleaner solution would involve updating `known_objects[...]["location"]` in `place()` itself (this is outside the scope here since only `agent.py` should be edited).
- With more time, I would add a short-term memory of *why* the robot declined or asked something, so a natural follow-up in the same conversation ("it's for cutting vegetables") could resume the initial task instead of starting anew each time. Currently, each `handle()` call is stateless between commands, which works fine for single test runs but not for a real back-and-forth conversation.
- I would also implement proper unit tests directly around `heuristic_action()` (instead of only testing via `run.py --test`), particularly for the word-boundary matching. For example, ensuring "glasses" is not incorrectly detected as "glass"/`empty_cup` - the substring-based version of this code originally had that bug, which is why matching is done with `\b` for word boundaries instead of plain substring checks.
